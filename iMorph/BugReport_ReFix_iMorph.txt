# Bug Report: iMorph fails after /reload or character logout/login

## Issue

After using `/reload` (UI reload) or logging out and back into a character, iMorph stops working. The chat message `iMorph Addon must be enabled. Options -> AddOns -> iMorph` appears, and morphing commands no longer function.

Sometimes it works after reload, sometimes it doesn't — this is non-deterministic.

## Root Cause Analysis

The issue is caused by a stale `lua_State*` pointer inside `iMorph.dll`.

### How it works currently

1. `iMorph.dll` is injected into the WoW process via `LoadLibraryA`
2. During `DllMain` (or initialization), the DLL obtains and caches the current `lua_State*` pointer from the WoW client
3. All subsequent communication from the Lua addon goes through `_G["CloseTrade"]()` → DLL → uses cached `lua_State*`

### What happens on /reload

1. `/reload` destroys the entire Lua environment (old `lua_State*` is freed)
2. A brand new `lua_State*` is created
3. All addons are reloaded (including `iMorph.lua`)
4. `_G["CloseTrade"]` binding is re-established in the new Lua environment
5. **BUT** the DLL still holds the pointer to the **old, freed** `lua_State*`
6. When the addon calls `CloseTrade()`, the DLL uses the stale pointer → undefined behavior
7. If the old memory hasn't been reused yet, it may appear to work (non-deterministic)
8. If the memory has been overwritten, it corrupts game state — including network modules, which causes chat disconnection

## Suggested Fix

The DLL should detect when the `lua_State*` becomes invalid and re-acquire it. Here's the approach:

```c
// Before processing any command from Lua, validate the cached lua_State*
// Option 1: Store the lua_State* pointer that the Lua callback receives
// (this is the CURRENT valid pointer) and compare with cached one

// In your CloseTrade hook / Lua callback:
void OnCloseTrade(lua_State* L) {
    if (L != g_cached_luaState) {
        // Lua state changed (e.g., after /reload)
        g_cached_luaState = L;
        // Re-register any Lua callbacks or references that depend on lua_State
    }
    // ... proceed with command handling
}

// Option 2: Before using g_cached_luaState, check if the memory is still valid
// by calling lua_gettop(g_cached_luaState) or similar and see if it crashes
// (less clean, but simpler)
```

The key insight is: **every time Lua calls into your DLL, the `lua_State* L` parameter in the callback IS the current valid pointer**. Use this to detect reloads and update your cached state.

## Environment

- WoW Version: 11.x (The War Within)
- iMorph.dll: 283,864 bytes (1.0.1)
- OS: Windows 10/11 x64
- Reproduced consistently across multiple sessions

## Additional Note

I've also observed that calling the DLL with a stale `lua_State*` can corrupt the game's network state, causing `"You have been disconnected from the server"` errors. Adding the validation above would also prevent this collateral damage.
